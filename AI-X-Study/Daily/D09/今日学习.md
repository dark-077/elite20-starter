## 🔄 迭代记录

### 第 1 次迭代

**问题描述**：
AI 摘要输出格式不稳定，有时返回纯文本，有时返回 JSON，难以程序化处理。

**修改内容**：
在提示词中添加严格的输出格式约束，并增加"如果无法解析，返回纯文本"的兜底逻辑。

**修改后的提示词**：
```
请严格按照以下 JSON 格式输出，不要包含任何其他内容：
{
  "title": "文章标题（20字内）",
  "summary": "摘要内容（100字内）",
  "highlights": ["要点1", "要点2", "要点3"],
  "conclusion": "一句话总结"
}

如果文章内容不足，字段可为空字符串，highlights 数组至少返回1个要点。
```

**结果**：✅ JSON 解析成功率从 60% 提升至 95%

---

### 第 2 次迭代

**问题描述**：
长文章（超过 2000 字）摘要质量下降，经常遗漏关键信息。

**修改内容**：
将长文拆分为多个段落分别摘要，再合并结果。同时增加"必须包含文中提到的具体数据"的要求。

**结果**：⚠️ 部分改善，但增加了一次 API 调用，成本上升

**成本分析**：
| 版本 | 平均响应时间 | Token 消耗 | 单次成本 |
|------|-------------|-----------|---------|
| v1.0 | 3.2s | ~800 | $0.002 |
| v2.0 | 5.1s | ~1500 | $0.004 |

---

### 第 3 次迭代

**问题描述**：
用户反馈"要点"部分过于笼统，需要更具体的 actionable 建议。

**修改内容**：
将"要点"改为"行动建议"，要求 AI 给出具体可执行的建议而非泛泛而谈。

**结果**：✅ 用户满意度显著提升，NPS 从 6 升至 8

---

### 第 4 次迭代（本次）

**问题描述**：
微信端响应速度慢，用户等待时间超过 10 秒。

**修改内容**：
1. 添加"正在分析中..."的即时回复
2. 引入流式输出（streaming）减少感知延迟
3. 优化工作流，减少不必要的等待

**结果**：✅ 感知响应时间降至 3 秒内

## 🐛 调试技巧总结

**学到的方法**：

1. **最小复现法**：遇到 Bug 时，先剥离所有非必要因素，在最小化场景中复现问题

2. **版本化提示词**：将每次提示词修改视为一个版本，保留历史记录便于回溯

3. **用户反馈驱动迭代**：不要猜测用户需求，用真实反馈指导改进方向

4. **成本-效益权衡**：每次迭代都要评估 API 成本 vs 用户体验提升

5. **渐进式增强**：先保证核心功能稳定，再逐步添加高级特性

**调试命令**：
```bash

# 测试提示词变体

# 检查 API 配额
coze usage --month current
```

## 💭 反思

**调试与迭代学习总结**：
> 这次练习让我深刻理解了"迭代"的力量。不是追求一次完美，而是在快速试错中持续优化。最让我印象深刻的是第 3 次迭代——用户反馈"要点太笼统"，这个问题我完全没有意识到，说明"闭门造车"式的优化是有局限的。

**关键教训**：
- AI 产品的调试需要同时关注"AI 行为"和"用户体验"两条线
- 提示词优化是渐进的，每次小改动的积累能带来显著提升
- 不要害怕引入成本——如果用户愿意为更好的体验付费，成本上升是合理的
